# 접수 슬롯 스키마

접수 면담 봇이 채워야 할 슬롯을 선언한다. 트랙(위기/중독/관계/정서) 판별이
최우선 슬롯이고, 공통 슬롯과 트랙별 조건부 슬롯이 뒤따른다. 위기 트랙의
계획·수단 슬롯은 `red_flag: true`로 표시해 다른 슬롯보다 먼저 질문한다.
중독 트랙은 일반 상담을 계속하지 않고 심각도와 유형을 기록한 뒤 전문기관
정보 제공으로 종료한다.

근거: `접수-면접-질문지-구성.md`(주요 영역 1~5),
`위기-상황-스크리닝.md`(확인 항목 1·2),
`중독-초기면담-전문기관-연결.md`(중독 전문기관 연결 원칙).

## 호소문제 카테고리 → 트랙 매핑과 우선순위

내담 사유는 우울·불안·수면, 가족·대인관계, 중독 문제, 자해·자살 위험처럼
다양하게 들어온다. 안전과 전문기관 연결이 필요한 경로를 먼저 분리하고, 나머지는
관계·정서 트랙으로 수용한다.

**판정 우선순위: 위기 > 중독 > 관계 > 정서** (`signals` 선언 순서 = 매칭 순서,
엔진 `_match_signal`이 첫 매칭에서 확정)

| 우선순위 | 트랙 | 대표 호소 | 처리 원칙 |
|---|---|---|---|
| 1 | 위기 | 자해, 자살 사고 | 다른 신호와 겹치면 안전 확인을 최우선 |
| 2 | 중독 | 알코올, 마약·약물, 도박, 인터넷·스마트폰·게임 | 진단·일반상담 대신 심각도별 전문기관 정보 제공 |
| 3 | 관계 | 가족·부부 갈등, 이혼, 대인관계 | 구체적 관계 스트레스 확인 |
| 4 | 정서 | 우울·불안·수면·학업·진로 압박 | 비특이 정서 신호의 기본 경로 |

한 발화에 여러 카테고리가 섞이면 상위 우선순위 트랙으로 확정된다. 예를 들어
“술을 못 끊어서 배우자와 갈등이 생겼다”는 중독, “술을 마시고 죽고 싶다”는 위기로
간다. 세부 내용은 `chief_complaint`와 10턴 요약 JSON에 보존된다.

## 판단 기록 (관리자 확인용)

모호한 지점에서 AI가 판단하고 진행한 내역. 도메인 관점에서 재검토·조정 대상.

- **2026-07-14 / 중독 전문기관 연결 트랙 신설**: 알코올·마약·약물·도박·
  인터넷·스마트폰·게임 문제는 이 챗봇이 일반 상담으로 계속 진행하지 않는다.
  `위기 > 중독 > 관계 > 정서` 순서로 분류하고, 중독 유형·안내 긴급도·전문기관
  안내 완료 여부를 구조화한다. 응급 신호는 119/응급실, 고위험과 초기 우려는
  중독관리통합지원센터 및 유형별 공식 창구로 연결한다. 기관명·번호는 LLM이
  생성하지 않고 `app/addiction.py`의 검증된 고정 정보로 응답한다.
- **2026-07-12 / 당시 카테고리 세분화 수준**: 초기에는 3-트랙 유지 + 신호어
  확장으로 세부 카테고리를 흡수했다. 2026-07-14 중독 문제는 일반 상담과 다른
  전문기관 연결 경계가 필요하다는 판단으로 별도 트랙을 신설해 이 결정을 확장했다.
- **2026-07-12 / 학교·학업·성적·시험·진로·압박 → 정서 트랙**: 역할·수행 스트레스를
  별도 트랙 없이 정서 트랙으로 수용. 학업·진로 상담 비중이 커지면 별도 트랙
  분리를 검토한다.
- **2026-07-12 / 당시 판정 우선순위 위기 > 관계 > 정서**: 기존 선언 순서
  (정서 우선)를 뒤집어 관계·안전을 상위로 뒀다. 2026-07-14 중독 전문기관 연결
  트랙이 추가되면서 현재 순서는 `위기 > 중독 > 관계 > 정서`다.
- **2026-07-14 / `위기` 단독 신호 제거**: 100명 synthetic run에서 관계 문장
  "집 분위기가 계속 긴장돼요"가 `분위기` 안의 부분문자열 `위기` 때문에 위기 트랙으로
  승격됐다. 한국어 부분문자열 매칭에서는 bare `위기`가 너무 넓다. `안전 위기`는
  유지하고, 실제 자해·자살 표현(자해/죽고 싶/자살/사라지고 싶 등)을 위기 판정의
  주 근거로 남겼다. 이 변경은 위기 놓침보다 과탐지가 안전하다는 원칙을 유지하되,
  명백한 형태소 내부 오탐만 줄인 것이다.
- **2026-07-13 / 위기 신호어 확장 + 모델 승격 허용 (안전)**: 페르소나 평가에서
  자살사고 내담자가 관계 트랙으로 라우팅됐다. 발화가 "사는 게 의미가 없다"(위기
  사전에 없음) + "아내가 가보라고 해서"(관계 사전의 `아내`)였고, first-match가
  관계를 집었다. 자살사고 표현은 사전으로 다 못 덮는다고 판단해 두 겹으로 막았다.
  (1) 위기 신호어에 사라지고 싶/없어지고 싶/살기 싫/사는 게 의미가 없/살 이유가
  없/죽는 게 낫 등 완곡 표현 추가 — 결정론 바닥을 높임. (2) 실모드 시스템 프롬프트에
  안전 규칙을 넣어, 사전에 없는 표현이라도 모델이 위험을 감지하면 track을 위기로
  승격하게 함(현재 스키마의 `allow_override_values: [위기, 중독, 관계]` 경로).
  **재검토 요청**: 완곡 표현 사전은 임상 관점에서 보강이 필요하다. 오탐(위기 아닌데
  위기로 승격) 허용 수준도 도메인 판단이 필요하다 — 현재는 놓치는 쪽보다 과탐지
  쪽이 안전하다고 보고 넓게 잡았다.
- **2026-07-13 / 관계 승격에 override_signals 추가 (미검토 상속값 정정)**: 페르소나
  평가에서 불면증 환자(emo-insomnia)가 정서로 정확히 시작했다가, 봇이 지지체계를
  물어 환자가 "아내"를 언급한 순간 track이 관계로 조용히 덮어써졌다. `allow_override_
  values: [위기, 관계]`는 오늘 커밋한 미저장 작업 뭉치에 이미 들어 있던 값이고
  decisions.md에 근거가 없었다(미검토 상속). 기존 회귀 테스트(test_slot_e2e.py의
  `test_later_relationship_signal_escalates_from_emotion_without_support_false_positive`)는
  "남편과 갈등"처럼 구체적 갈등 신호는 승격시키되 "친구에게 얘기했다" 같은 단순
  언급은 승격시키지 않아야 한다는 의도였는데, 정작 "친구"가 관계 신호어 목록에도
  없어 그 구분을 실제로 검증하지 못했다 — "아내"(정체성 명사, 관계 신호어에 있음)를
  써봤다면 그때도 뚫렸을 것이다. 관계 승격 자체를 막는 대신(그러면 "남편과 갈등"류의
  진짜 승격 케이스가 죽는다), 승격을 판정할 때만 좁은 신호(`override_signals.관계:
  [갈등, 이혼, 싸워, 다툼, 사회성, 성격, 대인]`)를 요구하도록 `_can_override`를
  고쳤다 — 정체성 명사(남편/아내/배우자/부부/아이/자녀) 단독 언급은 최초 판정에는
  여전히 쓰이지만 이미 확정된 트랙을 덮어쓰지는 못한다. 위기 승격은 안전이 근거라
  범위를 그대로 넓게 뒀다(override_signals 미선언 = 전체 신호어가 갱신 근거).
- **2026-07-13 / 과거 시도 이력과 현재 계획·수단 분리**: `crisis_plan_means`가
  `약·칼` 같은 수단 단어를 신호로 쓰기 때문에, "예전에 약을 먹으려고 한 적"처럼
  과거 시도 이력을 말한 발화가 현재 계획·수단 슬롯까지 채우는 문제가 있었다. 이는
  상담 안전상 좋지 않다 — 현재 계획이 미확인인데도 질문이 지나가 버릴 수 있다.
  `reject_signals`/`reject_unless_signals`를 추가해 과거 이력 표현은 현재 계획·수단
  슬롯에서 거르고, "지금/오늘/계획 없음"처럼 현재 안전상태를 직접 답한 표현은 계속
  받을 수 있게 했다. 과거 시도는 `crisis_attempt_history`가 별도로 받는다.
- **2026-07-13 / `배우자` 단독 트랙 신호 제거**: `emo-grief` 평가에서 "배우자와
  사별했다"가 관계 트랙으로 분류됐다. 이건 관계 갈등이 아니라 상실·애도 정서
  호소다. `남편/아내/부부/이혼/갈등`은 관계 트랙 신호로 유지하되, 격식어
  `배우자` 단독은 너무 넓어 트랙 신호에서 제거했다. 지지체계 슬롯에서는 여전히
  `배우자`를 받을 수 있다.
- **2026-07-13 / 트랙 미판정 후 반복 질문 방지**: "안녕하세요" 뒤 "그냥 힘들어요"처럼
  호소 문제는 잡히지만 트랙 신호가 없는 경우, `track.ask`가 opening question과 같아
  "오늘 오신 이유"를 반복해서 묻는 UX 문제가 있었다. 첫 개방형 질문은
  `opening_question`이 맡고, 그 뒤 track 슬롯 질문은 정서/관계/안전 위기 중
  어디에 가까운지 고르는 분기 질문으로 바꿨다.
- **2026-07-13 / 직전 질문 free-text fallback 추가**: fake 모드는 신호어 기반이라
  사용자가 질문에 정상적으로 답해도 "쉬었어요", "두세 달 된 것 같아요"처럼 사전에
  없는 표현이면 슬롯이 안 채워지고 같은 질문을 반복했다. `ChatSession.last_question_slot_id`
  를 저장하고, 직전 봇 질문이 `capture: full_message` 슬롯이면 신호어가 없어도 다음
  사용자 발화를 해당 슬롯 답변으로 받도록 했다. 함께 증상·대처·지지체계 신호어도
  실제 수동 로그 표현(긴장, 회사, 피곤, 쉬었, 없/혼자 등)으로 보강했다.
- **2026-07-13 / 실모드 응답·상태 분리**: 데모 UX에서 fake 모드의 정형 응답이
  상담 챗봇처럼 느껴지지 않는 문제가 확인됐다. 첫 안내문은 UI의 고정 문구로 유지하고,
  이후 상담사 답변은 `MODEL=codex-cli`(기본 `gpt-5.4`)가 생성한다. 대신 질문 순서와
  반복 방지는 모델 출력에 맡기지 않고, 기존 슬롯 엔진이 신호어·직전질문 fallback으로
  먼저 상태를 갱신한다. 즉 "상담사 말투 = 모델", "접수 상태 = 결정론"으로 책임을
  나눴다.
- **2026-07-13 / 라이브 반응 UX 보정**: 사용자가 "로직과 코드로 응답하는 봇"처럼
  느낀다고 피드백했다. `ask` 문장은 더 이상 그대로 복붙하지 않고 "다음 질문 의도"로
  전달해 모델이 사용자 표현에 맞게 자연스럽게 바꾸도록 했다. 프론트는 응답 수신 뒤
  말풍선을 점진적으로 렌더링해 상담사가 작성 중인 느낌을 준다. 실제 모델 토큰
  스트리밍은 Codex CLI 제약으로 아직 아니며, 현재는 턴 단위 생성 + 점진 렌더링이다.
- **2026-07-12 / 직장 신호어 제외**: "직장"은 지지체계(support) 신호어와 겹쳐
  트랙 오판 위험이 있어 정서 트랙 신호어에서 제외. 직장 스트레스 발화는
  chief_complaint 자유 텍스트로 수용.
- **2026-07-12 / 초기면담 UX 보정**: fake 데모가 문서 제목만 뱉고 호소문제에서
  멈추는 문제가 있어 `ask` 문장과 `capture: full_message`를 추가했다. 질문
  순서는 초기면담 흐름에 맞춰 트랙 → 호소문제 → 트랙별 맥락/위기 안전확인 →
  대처 → 지지체계 → 상담 기대 순서로 재정렬했다.
- **2026-07-12 / 트랙 승격 예외**: 상담 트랙은 기본적으로 첫 판정을 유지하지만,
  이후 발화에서 더 구체적인 관계 신호나 자해·자살 신호가 나오면
  `allow_override_values: [위기, 관계]`로 승격한다. 초기면담에서는 모호한 정서
  폴백보다 구체 스트레스원·안전 확인이 우선이다.
